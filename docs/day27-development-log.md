# Day27 开发日志

日期：2026-07-19

## 今日目标

在 Day26 完成功能包拆分之后，继续整理项目内部最容易产生长期隐患的“关系提示、事件记忆和用户档案”链路。今天的目标不是增加表面功能，而是建立一套可信、可追溯、可过期、可纠错的记忆治理机制，避免 AI 推测被当成用户事实保存，也让旧数据可以平滑迁移到新的结构。

## 完成内容

### 1. 统一关系行为提示逻辑

- 将关系阶段、熟悉度、亲密度和角色行为提示统一收敛到 `character/relationship_behavior.py`。
- 正常聊天、上下文构建和行为引擎现在共享同一套关系策略，不再分别维护相似但可能逐渐漂移的提示文本。
- `behavior_engine.py` 保留兼容导出，旧调用方不需要一次性改写。
- 以后调整 Mio 在不同关系阶段的距离感、措辞和主动程度时，只需维护一个权威入口。

### 2. 建立 Memory Record v2

- 为事件记忆增加统一的第二版记录结构，核心字段包括：
  - `type`：记忆类别；
  - `status`：当前生命周期状态；
  - `source`：事实来源；
  - `confidence`：可信度；
  - `evidence`：来自用户原话的证据；
  - `expires_at`：过期时间；
  - `updated_at`：最后更新时间；
  - Embedding 模型与版本信息。
- 支持 `identity`、`preference`、`plan`、`emotional_episode`、`relationship_moment`、`temporary_context` 等类型。
- 支持 `candidate`、`confirmed`、`legacy`、`superseded`、`expired`、`retracted` 等状态。
- 支持 `user_explicit`、`user_corrected`、`system_observed`、`ai_inferred`、`legacy_import` 等来源。
- 旧事件在读取时自动补齐为 `legacy_import + legacy`，不要求手动修改现有 JSON 数据。

### 3. 为事件增加事实来源约束

- 事件提取提示明确要求返回可在用户原话中验证的证据，而不是只返回 AI 自己总结的结论。
- 本地会再次检查证据是否真实存在于本轮用户消息中，形成提示约束之外的第二道防线。
- 缺少有效证据或仅由模型推测得到的内容会被标记为 `ai_inferred + candidate`。
- 候选记忆不会进入：
  - 对话上下文；
  - 语义检索；
  - 主动聊天；
  - 关系数值更新；
  - 角色长期状态。
- 候选记录默认在 7 天后过期，防止未经确认的推测长期堆积。
- 上下文与语义检索优先使用经过验证的用户证据，而不是未经约束的 AI 摘要。

### 4. 增加记忆类型、状态和过期机制

- 临时上下文默认保留 1 天。
- 情绪片段默认保留 7 天。
- 计划类记忆默认保留 30 天。
- 身份、稳定偏好等长期事实不套用通用短期过期时间。
- 已过期、撤回或被替代的记录不会继续参与检索和角色决策。
- 加载旧数据和执行迁移时会持久化补全后的状态，避免每次启动重复推断。

### 5. 建立稳定档案层

- 用户档案不再只是若干直接覆盖的顶层字段，而是增加带来源、状态、置信度、证据和历史的稳定事实结构。
- 旧版姓名、昵称、喜欢和不喜欢等字段会自动迁移为兼容事实，同时继续生成原有顶层字段供旧模块读取。
- 支持档案事实的新增、删除、合并、来源优先级和冲突覆盖。
- 用户明确纠正旧信息时：
  - 新事实使用 `user_corrected` 来源；
  - 旧事实被标记为替代或撤回；
  - 历史记录继续保留，方便以后审计和恢复。
- 档案提取同样要求新值真实出现在用户原话中，避免模型把补充想象写入稳定档案。

### 6. 为缺失 Embedding 增加后台回填

- 为事件向量增加独立的 Embedding 版本号。
- 启动后会扫描缺失或版本过旧的事件向量，并按小批次在后台回填。
- 回填计算在锁外执行，只在写入批次结果时短暂持锁，避免阻塞正常聊天和记忆读取。
- 每批结果使用现有原子写入机制保存，中途退出时不会破坏事件文件。
- GUI 会在语义模型预热、TTS 预热之后安排回填，降低多个本地模型同时抢占资源导致启动卡顿的风险。
- 控制台入口也接入相同的后台回填能力。

### 7. 增加记忆治理自动化测试

- 新增 `tests/test_memory_governance.py`，覆盖 16 个核心场景。
- 主要验证内容包括：
  - 旧事件向 Memory Record v2 迁移；
  - 明确用户事实与 AI 推测的来源区分；
  - 无证据候选记忆不会进入检索和关系更新；
  - 不同记忆类型的默认过期策略；
  - 已过期、撤回和被替代记录的过滤；
  - 稳定档案的新增、合并、纠正和冲突历史；
  - 缺失及旧版本 Embedding 的后台回填；
  - 关系行为提示只有一个权威实现。

## 关键问题与结论

### 为什么不能直接保存 AI 总结

AI 总结适合帮助组织语言，但不天然等于事实。只要模型补充了用户没有说过的细节，错误就可能经由长期记忆、主动聊天和关系系统不断放大。现在系统要求可追溯到用户原话的证据；无法验证的内容仍可短暂保留为候选，但不能影响角色长期行为。

### 为什么需要 type 与 status 两个维度

`type` 回答“这是什么记忆”，决定默认保留时间和使用场景；`status` 回答“这条记忆现在是否有效”，决定它能否进入检索和决策。只使用一个字段无法同时表达“这是一个计划，但已经过期”或“这是一个偏好，但后来被用户纠正”。

### 为什么稳定档案要保留冲突历史

直接覆盖虽然简单，却无法判断资料为什么发生变化，也无法在误提取后恢复。新档案层会保留来源、证据和被替代关系，让“用户以前喜欢 A，后来明确说不喜欢了”成为一次可解释的更新，而不是悄悄改写历史。

### 为什么 Embedding 回填放在后台

向量回填属于维护任务，不应阻塞窗口出现、首轮聊天或语音预热。分批后台处理可以逐步修复旧数据，同时把启动阶段的 CPU、内存和模型资源竞争控制在较低水平。
