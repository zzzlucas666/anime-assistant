# Day23 开发日志

日期：2026-07-15

## 今日目标

在 Style-Bert-VITS2 第一版语音已经能够稳定运行的基础上，对比并接入音色还原度更高的 GPT-SoVITS V2ProPlus；同时重构原有的情绪转换逻辑，让 Mio 不再只是根据长期事件切换几个固定标签，而是能够结合用户情绪、当前关系、回复语境和情绪强度做出更自然的反应，并保证文字、状态栏、Live2D 与 TTS 使用同一份状态。

## 完成内容

### 1. GPT-SoVITS V2ProPlus 对比与版本选择

- 对 Style-Bert-VITS2 与 GPT-SoVITS V2ProPlus 进行了固定文本试听对比。
- 确认 GPT-SoVITS 在当前授权 Mio 原声数据上的音色还原度更好，适合作为下一阶段主要语音后端。
- 在 RTX 4060 Laptop 8GB 显存条件下评估 V3 LoRA 与 V2ProPlus 的训练成本，最终优先采用已经稳定完成的 V2ProPlus 版本。
- 对多个训练检查点进行试听，选择 GPT e15 与 SoVITS e8 作为当前组合。
- 保留 Style-Bert-VITS2 后端，便于后续继续进行音色、稳定性和训练路线对照。

### 2. GPT-SoVITS 跨环境后台桥接

- 新增 `tts_gpt_sovits_worker.py`，在独立 GPT-SoVITS Python 环境中加载模型。
- 主程序继续使用 Python 3.14、PySide6 与 Live2D，避免直接安装训练环境中的深度学习依赖。
- 后台模型进程按需启动并常驻，减少每次回复重复加载权重的时间。
- 主进程与后台进程通过带请求 ID 的 JSON 行协议通信，避免多条任务结果串线。
- 对模型路径、参考音频路径和临时输出路径进行约束，生成完成后由主进程读取并清理 WAV。
- 保留完整回复级别的 AivisSpeech 原子回退，主后端失败时不会出现一句话中途切换音色。

### 3. 五种基础心情参考音频

- 为 `neutral`、`happy`、`shy`、`sad`、`tired` 分别配置 Mio 原声参考片段与准确日语提示文本。
- 新增 `scripts/generate_mio_emotion_samples.py`，可以使用固定测试句批量生成不同心情的对照样本。
- GPT-SoVITS 后端会根据当前角色心情选择对应参考音频，使语音、Live2D 和状态栏保持一致。
- 所有授权原声、训练数据、模型权重与试听输出继续保存在被 Git 忽略的 `data/` 目录，仅上传桥接代码和配置模板。

### 4. 情绪系统从“事件标签”升级为“反应状态机”

- 在生成回复前增加本地反应规划，不再等长期事件提取结束后才决定本轮心情。
- 回复完成后只允许校准已经存在的本轮反应，避免 Mio 因为自己上一句的害羞或开心语气而无限续期同一种情绪。
- 将用户情绪与 Mio 的反应分开：例如用户说自己难过时，用户状态为 `sad`，Mio 的反应为 `worried`，不会机械复制成 Mio 自己也低落。
- 持久主心情继续使用 `neutral`、`happy`、`shy`、`sad`、`tired`，同时新增短暂修饰情绪：
  - `worried`：担心、关心用户；
  - `touched`：受到触动或真诚感谢；
  - `curious`：短暂好奇；
  - `surprised`：短暂惊讶；
  - `annoyed`：轻微无奈或不满。
- 长期事件提取新增 `user_emotion`，并明确区分“事件中的用户情绪”和“Mio 对事件的反应”。

### 5. 夸奖反应与关系感知

- 外貌、可爱、声音或直接表达喜欢等个人夸奖，更容易触发害羞。
- 贝斯、唱歌、努力、进步或成果等能力夸奖，通常触发开心或感动。
- 如果 Mio 的实际回复出现结巴、不好意思或躲闪，最终状态会校准为害羞。
- 如果她自然接受夸奖并明确表达开心，状态可以校准为开心。
- 好感度、熟悉度与当前心情会参与判断：关系足够亲近时，直接表达喜欢不再永远只能得到固定害羞反应。

### 6. 情绪强度、持续轮数与自然退场

- 新增 `mood_strength`，同一种心情可以有轻微、中等和明显的表现差异。
- 新增心情持续轮数；无新触发时，心情会先按轮数保持，再逐步减弱并回到平静。
- 正向与负向心情之间的轻微信号需要连续确认，避免一条普通消息让状态突然翻转。
- 候选心情增加有效期；中间出现无关对话或等待过久后，旧候选不会继续累计。
- 短暂修饰情绪拥有独立强度和持续轮数，不会永久覆盖主心情。
- 用户情绪也会逐步淡出，不会因为一次分享而永久保留。

### 7. 疲惫状态双阈值与连续疲劳轴

- 将疲劳从单一精力阈值改为连续的 `fatigue_strength`。
- 设置不同的进入与退出阈值，避免精力在边界附近时反复切换“疲惫/非疲惫”。
- 修复精力恢复后仍可能卡在疲惫状态的问题。
- 回复提示、Live2D 和 TTS 统一读取疲劳强度，不再出现文字已经很累、表情和声音仍然开心的不同步。

### 8. Live2D 与状态栏联动

- `CharacterController` 不再只在心情名称变化时刷新；同一心情的强度发生变化也会重新计算目标参数。
- Live2D 情绪参数从中性值按 `mood_strength` 连续缩放，轻微开心不会直接使用满幅笑脸。
- 为担心、感动、好奇、惊讶和无奈增加眉毛、眼睛、嘴型与头部的轻量叠加参数。
- 疲劳程度会逐步影响眼睛、眉毛、嘴角和头部姿态。
- 状态栏可以显示“担心、感动、好奇、惊讶、无奈”等短暂状态，不再只能显示五种基础心情。
- 保留已有表情调参预设和全局强度设置，新的动态强度在现有预设基础上工作。

### 9. 回复语气与 TTS 强度联动

- system prompt 增加本轮即时反应提示，并明确要求先接住用户感受，不得输出内部标签或数值。
- 对担心、感动、好奇、惊讶和无奈分别增加简短自然的说话规则。
- 疲劳回复由连续强度控制，不再使用一个硬阈值突然变短。
- TTS 会根据主心情和强度选择参考音频；特别轻微的情绪可以使用中性参考，避免所有情绪都满强度输出。
- 在不明显改变现有音色的前提下，根据开心、害羞、低落、担心、惊讶和疲劳加入小幅连续语速调整。

### 10. 测试、配置与文档

- 扩展 `config_loader.py`、`data_models.py` 和 `settings_example.json`，校验 GPT-SoVITS 后端、权重、参考音频和新增情绪字段。
- 新增情绪状态机测试，覆盖用户难过、夸奖分流、关系感知、自我续期、候选过期、自然淡出、疲惫双阈值和担心事件。

## 关键问题与结论

### 为什么之前文字害羞，但状态栏仍然开心

旧逻辑把即时心情依附在“是否值得进入长期事件记忆”的判断上，而事件情绪枚举没有害羞。普通夸奖可能不被记录，即使回复文字已经表现得不好意思，状态也不会及时改变。现在即时反应和长期事件已经拆成两条通道，本地反应会在本轮 TTS 与 Live2D 开始前提交。

### 为什么不能直接把用户的情绪复制给 Mio

用户说“我很难过”描述的是用户本人。Mio 更符合人设的反应通常是担心、认真倾听或安慰，而不是把自己的主心情直接切成低落。新的 `user_mood + Mio mood + modifier` 结构可以分别表示双方状态。

### 为什么使用持续轮数而不只依赖时间

只按 20 分钟衰减会导致连续聊天时某个心情停留太久。持续轮数可以让害羞、惊讶和好奇在几轮内自然退场，同时保留真实时间衰减作为长时间离开后的兜底。

### 当前语音为什么仍有机械和克制感

GPT-SoVITS 已经明显提高音色还原度，但目前授权训练素材量和语气覆盖仍然有限。机械感主要来自日常语速、自然停顿和轻重音变化不足，不适合只靠增加训练步数解决。后续应优先补充准确转录、语气多样、停顿自然的素材，并为不同情绪准备轻微、中等和明显三个强度层次。
