# Day29 开发日志

日期：2026-07-20

## 今日目标

在 Day28 完成人格 Prompt 与情绪语音协同后，开始第二阶段工程化整理。本日不增加新的可见功能，重点降低配置层与运行时服务的耦合、限制日志文件增长、建立跨模块数据契约，并根据真实测试修复“语音模型加载中输入 `exit` 导致窗口卡顿”的关闭并发问题。

## 完成内容

### 1. 解除配置层对完整 TTS 服务的反向依赖

- 新增 `anime_assistant/speech/config.py`，集中保存 AivisSpeech、Style-Bert-VITS2 和 GPT-SoVITS 的纯配置默认值。
- 新增 `anime_assistant/ai/config.py`，将 OpenAI 兼容服务的默认地址与客户端实现分离。
- `infrastructure/config.py` 不再为了读取默认值而导入 `speech/service.py` 和 OpenAI 客户端。
- 只加载配置时不再连带导入 TTS、音频、模型或 OpenAI 运行依赖。
- `speech/service.py` 和 `ai/client.py` 继续重新导出原有常量，已有调用方与测试无需修改。

### 2. 建立可轮转的统一日志

- 将普通 `FileHandler` 替换为 `RotatingFileHandler`。
- 默认单个 `app.log` 最大 5 MB，并保留 3 份历史日志，避免长期运行后日志无限增长。
- 支持通过以下环境变量覆盖限制：
  - `ANIME_ASSISTANT_LOG_MAX_BYTES`；
  - `ANIME_ASSISTANT_LOG_BACKUP_COUNT`。
- 所有模块 logger 共享同一个文件处理器，避免 Windows 下多个处理器同时轮转同一文件产生重命名竞争。
- 保留原有日志格式、INFO 文件级别和 WARNING 控制台级别。

### 3. 为语音队列建立明确任务类型

- 新增不可变 `SpeechJob`，显式保存：
  - 文本；
  - mood；
  - 情绪强度；
  - modifier；
  - 疲劳强度；
  - voice style；
  - voice style 强度。
- 新增 `WarmupJob`，将模型预热任务与普通语音任务明确区分。
- TTS 后台队列不再依赖位置含义不明确的七元素元组。
- 保留旧二元素、五元素、六元素和七元素元组的兼容转换，降低渐进重构风险。

### 4. 为情绪信号建立完整类型契约

- 新增 `EmotionSignal`，统一 mood、modifier、voice style、用户情绪、持续轮数、来源、置信度和候选列表等字段。
- 新增 `EmotionCandidate`，明确本地混合情绪候选的 reaction、语气、分数、原因和用户状态。
- 为情绪信号构造器、主情绪设置、次级反应设置和语气设置补充类型标注。
- 继续返回普通字典，保证 Prompt、Orchestrator、TTS 和 Live2D 现有调用完全兼容。

### 5. 修复模型冷启动期间退出卡顿

- 真实 GUI 测试发现：顶部仍显示“语音模型加载中”时输入 `exit`，窗口可能长时间卡顿。
- 定位到本地 TTS 客户端在等待模型 `ready` 时持续占用进程状态锁；关闭流程也需要同一把锁终止子进程，因此可能被最长 180 秒的启动等待阻塞。
- 将“启动串行锁”与“进程状态锁”分离：
  - 启动尝试仍保持串行；
  - 等待模型完成时不再占用关闭锁；
  - 窗口关闭可以随时中断启动等待。
- 为本地 TTS 客户端增加立即取消路径，退出时直接终止正在加载或合成的子进程。
- 关闭期间不再触发后续语义模型预热，避免窗口已退出后继续启动高开销后台任务。
- Style-Bert-VITS2 与 GPT-SoVITS 共用相同的可中断关闭机制。

### 6. 增加 Day29 架构与并发回归测试

- 验证单独导入应用配置不会加载 `speech.service` 或 `openai`。
- 验证多个模块共享同一个轮转文件处理器。
- 验证旧语音队列元组能够转换为不可变 `SpeechJob`。
- 验证 `speak()` 新请求实际进入类型化队列。
- 验证 `EmotionSignal` 的运行时字典字段与类型契约一致。
- 模拟模型仍在加载时调用 `shutdown()`，确认关闭耗时低于 0.5 秒。
- 验证模型启动等待没有继续持有进程锁，并可由 `cancel()` 主动打断。

## 关键结论

### 为什么配置不能导入完整服务

配置是应用最底层、最早加载的模块。它如果从完整 TTS 服务读取常量，就会间接加载 OpenAI、音频处理和本地模型相关依赖，使原本简单的配置测试也受可选依赖影响。把默认值放入纯配置模块后，依赖方向恢复为“服务读取配置”，启动和测试边界都更稳定。

### 为什么要保留旧队列兼容

类型化任务能减少字段顺序错误，但一次性删除旧元组协议会扩大重构范围。新服务负责在消费端统一转换旧载荷，既建立明确的新契约，也允许现有测试和可能仍在运行的调用方平稳迁移。

### 为什么退出需要立即取消

模型冷启动属于可放弃的后台工作。用户已经要求退出时，继续等待权重加载完成没有价值，还会让窗口表现为未响应。关闭路径现在优先停止昂贵任务并释放 UI，而正常启动与合成仍使用原有超时、重试和降级策略。
