# Day25 开发日志

日期：2026-07-17

## 今日目标

在 Day24 已经完成即时情绪、独立 `voice_style` 和三种对话入口兼容的基础上，解决纯关键词规则在自然表达、含蓄语气和多种合理反应之间仍然偏死板的问题。目标是在不新增一次 AI 请求、不牺牲首字响应速度的前提下，将正常聊天升级为“本地候选规则 + 同轮 AI 语义校准”的混合情绪判断，并建立一套可长期维护的固定对话回归基线。

## 完成内容

### 1. 本地规则从单一结论升级为多候选评分

- 每轮回复前仍然优先执行本地判断，保证网络异常时也能立即得到安全可用的情绪结果。
- 同一条用户消息不再只能对应一个固定反应，而是可以生成多组带置信度的候选：
  - 个人夸奖可以同时保留 `shy + bashful` 和 `happy + warm`；
  - 用户焦虑可以同时保留关心与安慰两种相近表达；
  - 好消息可以在开心、感动和惊讶之间保留合理空间。
- 关系好感、熟悉度、当前心情和表达方式会改变候选排序，不再完全依赖硬阈值。
- 最终本地计划会记录候选、置信度、原因和决策来源，方便日志诊断与后续调参。

### 2. 使用同一次主回复完成 AI 语义校准

- 正常聊天的 system prompt 要求 AI 在自然回复末尾追加一个紧凑的内部控制标签：
  - 用户当前感受 `user_mood`；
  - Mio 本轮反应 `reaction`；
  - 当前一句的 `voice_style`；
  - 反应强度与判断置信度。
- 控制信息由本来就要执行的主对话请求一并生成，不再额外调用一次情绪分类 API。
- 相比“回复完成后再调用 AI 判断情绪”，该方案只增加很少的尾部输出 token，不增加新的网络往返和线程等待。
- AI 可以结合完整语义区分同一句夸奖更像开心还是害羞，也可以理解关键词没有覆盖到的自然表达。

### 3. 流式隐藏标签与输出隔离

- 新增跨流式分块过滤器，即使 `<mio:...>` 标签被拆成多个 chunk，也能在进入界面前完整截获。
- 内部标签不会：
  - 显示在 GUI；
  - 写入聊天历史；
  - 进入长期记忆；
  - 被 GPT-SoVITS 或 AivisSpeech 读出来。
- `clean_reply()` 增加最后一道防御性清理，即使上游过滤异常也不会让控制标签污染可见回复。
- 主动聊天不使用这套交互式标签，继续采用已经验证的主动情绪规划，避免把不存在的当前用户输入误判为用户情绪。

### 4. 白名单、置信度与负面情绪保护

- 标签中的用户情绪、Mio 反应和声音风格必须属于项目已有白名单。
- 强度和置信度必须位于 `0.0～1.0`，低于最低置信度的结果会被直接忽略。
- 标签缺失、格式错误、值越界或流式输出中断时，系统自动保留本地候选结果，不影响文字回复。
- 明确的难过、焦虑、孤独、压力、疲惫和受挫属于本地高置信度保护场景：
  - AI 可以在 `concerned`、`reassuring`、`serious` 等同方向语气中细化；
  - 不能把需要关心的用户误校准成 Mio 的开心、害羞或轻快语气。
- 用户直接对 Mio 表达讨厌或失望时，同样保留本地失落反应，防止 AI 为了迎合而错误转成玩笑。

### 5. 情绪决策链路统一

- 正常对话最终依次执行：
  1. 本地候选规划；
  2. 可见回复措辞校准；
  3. 同轮 AI 控制信息安全校准；
  4. 原子提交状态；
  5. 更新 Live2D 与本轮 TTS。
- `ConversationOrchestrator` 会记录本轮决策来源、本地与 AI 置信度、用户情绪、Mio 心情、短暂 modifier 和 voice style。
- 后台事件提取仍然只负责长期记忆，不会反向覆盖已经开始播放的当前语音。

### 6. 固定对话回归基线

- 新增 `tests/emotion_dialogue_regression_cases.json`，保存人工确认的固定对话数据。
- 每个样例包含：
  - 用户原话；
  - Mio 可见回复；
  - 内部控制标签；
  - 初始心情与关系状态（需要时）；
  - 最终预期的用户情绪、Mio mood、modifier、voice style 和决策来源。
- 当前共保留 11 个核心场景：
  - 用户难过；
  - 考试焦虑；
  - 身体疲惫；
  - 夸奖后害羞；
  - 夸奖后自然开心；
  - 亲密关系中的直接好感；
  - 能力夸奖；
  - 普通偏好问题；
  - 惊喜好消息；
  - 对 Mio 的直接负面表达；
  - 错误 AI 开心覆盖被安全规则拒绝。
- 新增参数化回归测试，按真实生产顺序执行本地规划、回复校准、标签解析和 AI 安全校准，而不是只测试单个内部函数。

### 7. 回归样例发现的自然语言漏判

- 第一轮固定样例测试发现，`“我喜欢你”` 可以识别，但更自然的 `“我真的很喜欢你”` 因中间加入程度副词而没有命中固定短语。
- 没有通过修改测试句来绕过问题，而是为直接好感表达加入受约束的正则识别。
- 现在可以覆盖“我真的很喜欢你”“我很喜欢你”“我一直喜欢你”“我越来越喜欢你”等常见变化，同时避免把普通的“喜欢某个东西”误判成对 Mio 的直接表达。
- 该问题验证了固定对话回归集的价值：它不仅保护已有行为，也能暴露真实口语与关键词列表之间的缝隙。

### 8. README 与架构文档同步

- README 增加混合情绪系统说明，明确本地候选、同请求 AI 校准、隐藏标签、安全回退和负面情绪保护。
- 架构文档更新正常聊天时序，记录控制标签从生成、过滤到最终状态提交的边界。
- 增加回归基线维护规则：只有产品预期确实改变并经过人工试听或视觉确认后，才更新固定样例。

## 关键问题与结论

### 为什么不继续只增加关键词

关键词适合识别明确的“难过”“紧张”“好累”，但很难处理含蓄表达、程度副词、否定结构和同一句话存在多种合理反应的情况。继续扩大关键词表会让规则之间的优先级越来越难维护。当前方案保留关键词作为快速、安全的底线，同时让 AI 只负责语义细化。

### 为什么不单独再调用一次 AI 情绪分类

单独分类会增加一次网络往返、额外失败点和对话线程等待。将紧凑控制标签附在主回复末尾后，AI 已经拥有完整上下文，只需多输出少量结构化信息，首字显示仍然保持流式，最终情绪只在回复结束时提交。

### 为什么不能完全相信 AI 标签

模型可能遗漏标签、输出无效值，也可能在用户明显难过时给出过于积极的反应。因此本地候选不是临时过渡方案，而是始终存在的安全层。AI 只能在白名单和保护规则允许的范围内校准，不能绕过明确的用户痛苦信号。

### 为什么固定回归样例需要独立 JSON

把样例与测试代码分开后，可以直接阅读和评审对话本身，也方便未来增加真实聊天中发现的边界场景。规则、提示词或声音映射变化时，测试会明确指出哪一段角色行为发生了漂移。

## 后续方向

- 将已经稳定的情绪候选、策略校准和状态转换逐步拆分为职责更清晰的模块，避免 `emotion_manager.py` 继续膨胀。

- 为事件记忆增加价值过滤、相似事件合并和长期归档策略。
- 在准备网页、手机、OBS 或其他渲染客户端时，先建立统一内部事件格式，再接入 WebSocket 网关。
- 继续把真实聊天中发现的情绪边界加入固定回归集，使后续优化建立在可重复比较的基线上。
