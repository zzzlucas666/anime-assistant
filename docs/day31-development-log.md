# Day31 开发日志

日期：2026-07-21

## 今日目标

在 Day30 完成普通对话本地路由、状态治理和记忆边界收紧后，开始降低核心模块的维护成本。Day31 聚焦情绪管理与语音服务的纵向拆分，在不改变现有对话、情绪和 TTS 行为的前提下，把规则、状态、后端和调度职责分离，并保留原有导入入口，避免影响 UI 与其他业务模块。

## 完成内容

### 1. 纵向拆分情绪管理模块

- 将情绪关键词、正则表达式和检测辅助函数移动到 `emotion/rules.py`。
- 将普通对话、主动聊天和启动问候的候选评分与情绪规划移动到 `emotion/planning.py`。
- 将 AI 情绪结果的可信度控制、冲突保护和校准逻辑移动到 `emotion/calibration.py`。
- 将情绪加载、保存、持续轮数、恢复和状态转换移动到 `emotion/state.py`。
- `emotion/manager.py` 缩减为稳定兼容入口，继续导出旧调用方使用的公开函数。

### 2. 纵向拆分语音服务模块

- 将 AivisSpeech、Style-Bert-VITS2 和 GPT-SoVITS 客户端移动到 `speech/backends.py`。
- 将中文台词转日语的语音专用翻译器移动到 `speech/translator.py`。
- 将情绪到有效 mood、voice style 和语速倍率的映射移动到 `speech/style.py`。
- `speech/service.py` 只保留语音任务队列、预热、重试、备用后端和结果回调等协调职责。
- 保留旧的客户端、错误类型和静态辅助函数导出，现有 UI、测试与外部调用无需同步迁移。

### 3. 统一本地 TTS 工作进程生命周期

- Style-Bert-VITS2 与 GPT-SoVITS 共用进程创建、输出读取、启动等待、取消和关闭框架。
- 不同语音后端通过配置钩子提供依赖文件、环境变量、启动命令和错误类型。
- 保留加载期间可取消与退出的行为，避免重新引入 Day29 已修复的关闭卡顿。
- 语音拆分只调整代码边界，没有更改模型、权重、参考音频、超时或合成参数。

### 4. 增加架构兼容性回归

- 新增 `test_day31_architecture.py`，验证情绪管理兼容入口仍指向拆分后的真实实现。
- 验证语音服务仍能导出原有后端客户端、翻译器和辅助函数。
- 更新进程模拟测试的补丁位置，使测试对准新的后端模块边界。
- README 与架构文档同步记录拆分后的职责和兼容策略。

### 5. 调查 GPT-SoVITS 冷启动时间波动

- 实际测试中一次本地语音预热耗时约 98.9 秒。
- 日志显示其中约 91 秒用于首次导入 GPT-SoVITS 与 PyTorch 推理依赖，模型权重加载约 7 秒。
- 历史冷启动记录约在 20.9 至 89.9 秒之间，说明启动时间原本就会受 Windows 文件缓存、磁盘读取和安全扫描影响。
- 缓存变热后的独立计时中，PyTorch 导入约 2.35 秒，GPT-SoVITS 核心模块导入约 6.5 秒。
- 对比拆分前后启动命令和预热顺序，没有发现重复进程或重复模型加载，因此没有为单次冷启动波动引入高风险改动。

## 架构变化

```text
emotion.manager（兼容入口）
├── rules          规则与文本信号
├── planning       候选评分与本轮规划
├── calibration    AI 结果校准
├── state          持久化与状态转换
└── signals        跨模块情绪枚举

speech.service（任务协调）
├── backends       TTS 后端与本地进程
├── translator     日语语音翻译
├── style          情绪、语气与语速映射
├── text            可发音文本处理
├── audio           音频合并与口型包络
└── jobs            队列任务数据结构
```

## 实质提升

- 情绪规则、状态恢复和 AI 校准可以分别演进，降低修改一处误伤整套情绪系统的概率。
- 新增或替换语音后端时，不再需要继续扩大语音队列协调器。
- UI 与业务模块继续依赖稳定入口，内部重构不会迫使整个项目同步改写。
- 后续 Day32 可以在明确的服务边界上接入 `ApplicationRuntime` 与统一任务监督器。
- 独立模块更容易编写针对性测试、记录性能指标并定位模型加载或情绪判断问题。
