# Day32 开发日志

日期：2026-07-21

## 今日目标

在 Day31 完成情绪与语音模块纵向拆分后，继续收拢应用级生命周期和异步任务边界。Day32 的重点是引入统一 `ApplicationRuntime`、任务监督器和 `turn_id`，解决 GUI 与控制台重复创建服务、后台线程缺少统一治理，以及迟到回复、主动消息、后处理和语音可能影响新一轮交互的问题。

## 完成内容

### 1. 建立统一 ApplicationRuntime

- 新增 `anime_assistant/runtime/application.py`。
- Runtime 统一创建并持有对话编排器、主动聊天引擎、语音服务、上下文和共享状态。
- GUI 与控制台入口不再分别复制服务初始化、后台线程启动和关闭代码。
- 语音预热、语义模型预热和 Embedding 回填继续保持原有启动顺序，但由 Runtime 统一调度。
- 关闭程序时，由 Runtime 按依赖关系停止主动聊天、语音、对话后处理和其他受监督任务。

### 2. 增加单调递增的 turn_id

- 新增 `TurnCoordinator` 和不可变 `TurnContext`。
- 启动问候、用户输入和主动消息都会获得唯一轮次编号，例如 `turn-00000001-user`。
- 当前轮次是所有异步结果能否提交和展示的统一依据。
- 对话流、GUI 更新、主动消息、语音任务和音频数据均完成 `turn_id` 贯通。

### 3. 增加统一任务监督器

- 新增 `TaskSupervisor`、`TaskToken`、`TaskHandle` 和任务快照。
- 后台任务记录任务名、作用域、所属轮次、开始时间、结束时间和运行状态。
- 支持按任务、作用域或过时轮次发送协作取消信号。
- 对话流、顺序后处理、主动消息生成、TTS worker、语音合成、语义模型预热和 Embedding 回填接入监督器。
- 保留兼容路径，模块在脱离 Runtime 单独测试或调用时仍可自行工作。

### 4. 防止旧轮次污染新状态

- 新用户轮次开始后，旧回复流会停止继续向界面输出。
- 过时的 `finalize_turn` 不再提交即时情绪、关系或上下文。
- 迟到的后台事件仍可在事实校验后保存为长期记忆，但不能覆盖当前档案、情绪、关系和 Context。
- 主动消息在 AI 生成期间若被用户新输入抢占，会放弃迟到结果，不再插入当前对话。

### 5. 完善语音轮次隔离

- `SpeechJob` 与 `SpeechAudio` 增加可选 `turn_id`，旧元组任务继续兼容。
- 语音进入队列、分句合成、WAV 合并和 GUI 播放前都会校验当前轮次。
- 已经开始推理的旧语音允许自然返回，返回后静默丢弃，避免强杀常驻 GPT-SoVITS 模型。
- 轮次过期不会被当作 TTS 后端故障，不会触发备用后端、错误状态或本地模型重启。
- 新轮次开始时，GUI 会停止上一轮已经播放或排队的音频，避免旧台词在新回复期间继续发声。

### 6. 统一 GUI 与控制台入口

- `main_window.py` 只保留界面、Qt 信号和表现层职责，服务生命周期交给 Runtime。
- `ChatWorker` 显式接收并传递 `turn_id`。
- 主动消息和音频回调在进入 GUI 主线程后再次校验轮次。
- 控制台入口改用同一套 Runtime、问候规划、主动聊天和关闭流程，减少两种入口的行为分叉。

### 7. 文档与测试

- README 增加 Runtime 包职责、任务监督和轮次隔离说明。
- 架构文档补充运行时层、正常对话轮次协议和并发不变式。
- 新增 `tests/test_day32_runtime.py`，覆盖：
  - 轮次编号单调递增；
  - 旧任务协作取消；
  - 旧语音静默丢弃；
  - 轮次过期不重启 GPT-SoVITS；
  - Runtime 单例式启动和幂等关闭；
  - 旧事件只保存事实、不修改当前状态。

## 验证结果

- Python 3.14 全包语法检查通过。
- Runtime、SpeechService 和 GUI 主窗口导入烟雾检查通过。
- 语音、对话并发、主动聊天和 Runtime 专项回归通过。
- 全量自动化测试共 **203 项，全部通过**。
- 人工测试确认启动、聊天、语音、轮次切换和关闭流程稳定。

## 架构收益

1. 应用服务第一次拥有统一、明确的创建和销毁边界。
2. `turn_id` 将文字、状态、主动消息和语音组织为同一个可验证事务边界。
3. 后台任务由“零散线程”升级为可观察、可取消、可诊断的受监督任务。
4. 迟到结果只能完成安全的持久化工作，不能再破坏用户正在看到和听到的当前轮次。
5. 后续接入 WebSocket 时，可以复用 Runtime 和轮次协议，只增加新的输入输出适配器，不需要侵入 UI、对话或语音核心。
